0024: Индивидуальное задание на практику
0025: Целью практики является разработка и интеграция завершающих компонентов интеллектуальной системы управления паркингом, включая сервис аналитики, комплексное тестирование, оптимизацию производительности и подготовку выпускной квалификационной работы (ВКР). Основное внимание уделяется обеспечению стабильности системы, её масштабируемости и подготовке итоговой документации.
0026: Для достижения поставленной цели требуется решить следующие задачи:
0027: Разработка сервиса аналитики (analytics-service).
Необходимо реализовать микросервис, предоставляющий REST API для расчёта ключевых показателей эффективности парковки: текущая заполняемость, средняя длительность парковки, часы пик, utilisation мест, частота проездов. Все метрики должны вычисляться на основе данных из tracking_table PostgreSQL и агрегироваться через эффективные SQL-запросы.
0028: Интеграция аналитики с пользовательским интерфейсом.
Требуется подключить разработанный бэкенд аналитики к существующему фронтенду на React/TypeScript, обеспечив отображение реальных, а не моковых данных на дашбордах, графиках загрузки и тепловых картах.
0029: Проведение интеграционного и нагрузочного тестирования.
Необходимо выполнить интеграционное тестирование всех микросервисов системы (parking-management, cv-processing, stream-ingest, analytics), проверить корректность межсервисного взаимодействия и API-контрактов. Также следует провести нагрузочное тестирование ключевых эндпоинтов для выявления узких мест.
0030: Исправление выявленных критических ошибок.
Требуется устранить обнаруженные в ходе тестирования дефекты: рассинхронизацию счётчика available_spots, несоответствие типов SpotType и CameraStatus между фронтендом и бэкендом, а также реализовать механизм JWT-аутентификации для защиты API.
0031: Настройка брокера сообщений для асинхронной обработки.
Необходимо развернуть брокер сообщений (RabbitMQ/Kafka) для организации асинхронного взаимодействия между stream-ingest и cv-processing сервисами. Требуется настроить очереди на детекцию, трекинг, реализовать паттерн «Producer-Consumer», а также настроить dead-letter очереди и мониторинг состояния очередей.
0032: Оптимизация производительности и масштабирование.
Необходимо провести оптимизацию времени отклика API (цель — улучшение на 30%), внедрить кэширование часто запрашиваемых данных с использованием Redis, настроить индексы в PostgreSQL для ускорения запросов. Также следует сконфигурировать балансировку нагрузки (Nginx/HAProxy) и разработать рекомендации по горизонтальному масштабированию сервисов обработки видео.
0033: Подготовка пояснительной записки ВКР.
По итогам всех выполненных работ необходимо оформить пояснительную записку выпускной квалификационной работы в соответствии с требованиями ГОСТ. Документ должен включать введение, обзор аналогов, описание архитектуры системы, детали реализации ключевых модулей (CV-трекинг, аналитика, API, брокер сообщений, мониторинг), результаты тестирования и заключение.

0054: Актуальность задачи управления паркингом
0055: Автоматизированные системы управления паркингом (Parking Management Systems, PMS) представляют собой комплексные программно-аппаратные решения, интегрирующие средства компьютерного зрения, управления базами данных и коммуникационные протоколы. Основной задачей подобных систем является обеспечение непрерывного контроля за состоянием парковочных мест, отслеживание въезда и выезда транспортных средств, управление бронированиями и предоставление аналитической информации операторам.
0056: Рост числа автомобилей в городах и увеличение нагрузки на парковочную инфраструктуру формируют устойчивый спрос на интеллектуальные PMS. По данным аналитических исследований, неэффективное использование парковочных мест приводит к значительным потерям времени водителей и увеличению загруженности дорог. Автоматизация процессов учёта и управления позволяет увеличить пропускную способность парковочных объектов и снизить операционные издержки.
0057: В контексте темы ВКР ключевой технической задачей является не просто фиксация занятости мест, но интеллектуальное сопровождение конкретных транспортных средств в распределённой сети видеокамер — то есть корректная привязка наблюдаемого объекта к записи в базе данных при смене поля зрения камеры.
0058: Обзор существующих решений
0059: Для определения места разрабатываемой системы в ряду существующих решений был проведён анализ нескольких категорий продуктов.
0060: Hikvision DS-TCG225 и аналогичные ANPR-системы представляют собой аппаратно-программные комплексы, ориентированные преимущественно на распознавание номерных знаков. Их ключевые ограничения: жёсткая привязка к проприетарному оборудованию, отсутствие открытого API для интеграции с внешними системами и высокая стоимость лицензирования. Расширяемость подобных решений существенно ограничена.
0061: ParkMaster и аналогичные отечественные системы обеспечивают функции учёта и тарификации, однако не предоставляют инструментов компьютерного зрения и не поддерживают распределённую сеть камер с трекингом объектов между ними.
0062: Открытые решения на базе OpenCV и YOLOv5/v8 (например, репозитории типа vehicle-parking-detection) предоставляют базовую детекцию транспортных средств, но не решают задачу межкамерного Re-Identification и не имеют развитой серверной части для управления бронированиями, аналитики и аутентификации.

0068: Обоснование выбора микросервисной архитектуры
0069: Монолитная архитектура при разработке подобных систем сопряжена с рядом ограничений: сложность независимого масштабирования ресурсоёмких компонентов (например, модуля обработки видео), затруднённое параллельное развитие командой и высокие риски при развёртывании обновлений.
0070: Микросервисный подход позволяет изолировать каждый функциональный домен в отдельный сервис с собственной базой данных и жизненным циклом. В разрабатываемой системе выделены следующие домены: аутентификация и авторизация, управление парковочными объектами и бронированиями, обработка видеопотоков, аналитика, управление устройствами. Взаимодействие между доменами осуществляется через REST API (синхронно) и брокер сообщений RabbitMQ (асинхронно).
0071: Выбор языка Python (FastAPI) для бэкенд-сервисов обусловлен богатой экосистемой библиотек в области компьютерного зрения и машинного обучения (OpenCV, Ultralytics YOLO, torchreid), а также высокой производительностью асинхронных фреймворков при работе с I/O-bound задачами. Выбор React/TypeScript для фронтенда определяется развитой экосистемой и строгой типизацией, снижающей число ошибок при разработке.
0072: Описание программного продукта
0073: Общие сведения о системе
0074: Разрабатываемый программный продукт представляет собой интеллектуальную распределённую систему управления паркингом на основе компьютерного зрения. Система предназначена для автоматического отслеживания транспортных средств на парковочных объектах с использованием сети видеокамер, управления бронированиями и предоставления аналитики в режиме реального времени.
0075: Система реализована по микросервисной архитектуре и развёртывается в контейнерах Docker. Каждый компонент изолирован и взаимодействует с другими через REST API и брокер сообщений RabbitMQ. Все сервисы объединены единым шлюзом на базе Nginx.

0101: Проектирование модели данных выполнено с соблюдением третьей нормальной формы. Для ускорения запросов аналитики на таблице tracking_table создан составной индекс по полям timestamp и vehicle_id. Поле spot_coordinates таблицы spots_table хранит JSON-полигон разметки места в формате {"points": [[x,y],...], "center_x": float, "center_y": float}, что позволяет CV-модулю выполнять пространственное соответствие между обнаруженным объектом и конкретным местом на схеме парковки.
0102: Реализация программного продукта
0103: Parking-management-service
0104: Архитектура Router → Service → DAO
0105: Сервис управления паркингом организован по трёхслойной архитектуре. Слой Router отвечает исключительно за маршрутизацию HTTP-запросов и валидацию входных данных с помощью Pydantic-схем. Слой Service реализует бизнес-логику: проверку ограничений, оркестрацию операций с несколькими сущностями, публикацию событий в брокер. Слой DAO (Data Access Object) инкапсулирует все обращения к базе данных через SQLAlchemy AsyncSession.
0106: Управление жизненным циклом сессий базы данных было вынесено из слоя зависимостей (dependencies.py) непосредственно в классы DAO. Каждый DAO-класс получает сессию через контекстный менеджер with async_session() as session, что гарантирует закрытие соединения даже при возникновении исключений. Это устранило критическую проблему утечки соединений при write-операциях.
0107: Сервис содержит следующие роутеры: parking_router, spot_router, booking_router, vehicle_router, camera_router, analytics_router, events_router, internal_cv_router, mobile_compat_router, health_router.
0108: На рисунке 3 показана UML-диаграмма последовательности для операции создания бронирования.
0110: Сервис бронирований
0111: BookingService реализует следующую логику при создании бронирования: проверка существования парковочного места (404 при отсутствии); проверка статуса места (409 если OCCUPIED); проверка временного конфликта с существующими бронированиями через get_conflicting_bookings (409 при конфликте); создание записи бронирования; перевод места в статус RESERVED; синхронизация счётчика available_spots парковки; публикация события в RabbitMQ для обновления проекции.
0112: Жизненный цикл бронирования включает три статуса: PENDING (создано, ожидает подтверждения), CONFIRMED (подтверждено), CANCELLED (отменено с указанием причины). При отмене бронирования статус места возвращается в FREE с пересинхронизацией счётчика доступных мест парковки.
0113: Сервис аналитики
0114: AnalyticsService вычисляет следующие показатели на основе данных таблиц spots_table и tracking_table:
0115: occupancy_rate — отношение числа занятых мест к общему числу мест;
0116: avg_duration_minutes — среднее время стоянки, вычисляемое через сопоставление событий enter и exit для каждого vehicle_id;
0117: peak_hour — час суток с наибольшим числом событий въезда;
0118: hourly_traffic — массив из 24 элементов HourlyTrafficPoint{hour, vehicles} для текущих суток;
0119: weekly_occupancy — массив из 7 элементов WeeklyOccupancyPoint{day, occupancy%} за последние 7 дней;
0120: duration_distribution — распределение по бакетам: < 1 ч, 1–3 ч, 3–6 ч, > 6 ч;
0121: recent_events — последние 7 событий tracking_table с номером ТС и типом действия;
0122: mini_spots — статус первых 15 мест с привязкой к номеру ТС.
0123: Все метрики вычисляются за один SQL-запрос к tracking_table с JOIN на cameras для фильтрации по parking_id и JOIN на vehicles для получения plate_number. Временной горизонт запроса — последние 7 дней (start_week = now - 6 days).
0124: Обработка CV-событий
0125: Класс CVEventService потребляет сообщения из exchange cv.events и обрабатывает четыре типа событий:
0126: vehicle.detected / vehicle.track_started — запись наблюдения CVSpotObservation без изменения статусов;
0127: spot.occupied_confirmed — перевод места в статус OCCUPIED, привязка транспортного средства (vehicle.is_inside = True), запись в tracking_table с event_type='park', создание SystemEvent;
0128: spot.free_confirmed — перевод места в статус FREE, сброс current_vehicle_id, запись в tracking_table с event_type='leave_spot'.
0129: При обработке события spot.occupied_confirmed сервис выполняет автоматическое создание записи транспортного средства в базе данных, если оно ещё не зарегистрировано (по plate_number), что обеспечивает целостность данных при первом появлении нового ТС в системе.
0132: Auth-service
0133: JWT архитектура
0134: Auth-service реализует JWT/stateful-token архитектуру с четырьмя видами токенов. Класс TokenHandler параметризуется типом токена и извлекает из конфигурации алгоритм (HS256 для access/refresh, HS256 для invite/reset), секретный ключ и время жизни.
0136: Таблица 6 – Виды токенов auth-service
0138: Payload access-токена содержит поля: sub (user_id), role, type, exp, iat, jti (UUID для предотвращения повторного использования). Refresh-токен хранится в БД в виде хэша, что позволяет реализовать явный отзыв токена (revocation) без необходимости хранить чёрный список на стороне ресурсного сервера.
0139: Система ролей и разрешений
0140: Пользователи системы делятся на три роли: ADMIN (полный доступ), MANAGER (управление парковкой), TENANT (арендатор, доступ к бронированиям). Помимо ролевой модели реализована гранулярная система разрешений через таблицу permissions с кодовыми именами (code_name). Связь пользователь–разрешение хранится в таблице user_permissions (M:N). При инициализации сервиса функция bootstrap_defaults создаёт учётную запись администратора по умолчанию.
0141: На рисунке 4 показана UML-диаграмма классов auth-service.
0143: Интеграция брокера сообщений RabbitMQ
0144: В системе реализованы два независимых потока событий через RabbitMQ. Оба брокерных класса (BookingEventBroker и CVEventBroker) используют библиотеку aio-pika с надёжным подключением (connect_robust), которое автоматически восстанавливается при разрыве соединения.
0146: Таблица 7 – Конфигурация exchanges RabbitMQ
0148: Паттерн проекции бронирований (booking_projection) решает проблему N+1 запросов при отображении списка бронирований на фронтенде. Вместо выполнения JOIN-запросов в момент чтения, данные денормализуются при записи: при создании или обновлении бронирования сервис публикует событие в exchange booking.events, consumer обогащает его данными о пользователе и месте и записывает в таблицу booking_projection. Фронтенд читает уже готовую проекцию через эндпоинт GET /bookings.
0149: На рисунке 5 представлена UML-диаграмма последовательности для потока обработки CV-события.
0151: Device-emulator-service
0152: Device-emulator-service реализует программную эмуляцию внешних устройств парковочного объекта. Сервис предоставляет REST API для управления двумя типами устройств: шлагбаум (barrier) и освещение (lighting).

0199: Методология тестирования
0200: Тестирование разработанного программного продукта проводилось на двух уровнях: модульное (unit) тестирование сервисного слоя и интеграционное (e2e) тестирование роутеров. Изоляция зависимостей достигалась через AsyncMock из стандартной библиотеки unittest.mock, что позволило тестировать бизнес-логику без обращения к реальной базе данных.
0201: Архитектурный принцип тестируемости реализован через внедрение зависимостей в конструктор сервисного класса: все DAO-объекты и брокер событий передаются через __init__, что позволяет подменять их заглушками в тестах. Для e2e-тестов использован httpx.AsyncClient с ASGITransport, позволяющий выполнять HTTP-запросы к FastAPI-приложению без запуска реального сервера.

0218: Рефакторинг parking-management-service
0219: Сервис управления паркингом приведён к трёхслойной архитектуре: Router → Service → DAO. Слой DAO (Data Access Object) инкапсулирует все обращения к базе данных; слой Service содержит бизнес-логику; слой Router отвечает только за маршрутизацию HTTP-запросов и валидацию входных данных.

0227: Устранена проблема управления жизненным циклом сессий базы данных. 0228: Внедрение RabbitMQ
0229: В инфраструктуру проекта интегрирован брокер сообщений RabbitMQ. При создании или обновлении бронирования parking-management-service публикует событие через модуль rabbitmq.py в exchange booking.events с routing key booking.changed. На старте сервис поднимает consumer, который читает очередь booking.projection и передаёт данные в booking_projection_service.py. Этот сервис обновляет таблицу-проекцию, которую читает фронтенд через эндпоинт /bookings, получая обогащённые данные (имя пользователя, номер места, статус) без дополнительных JOIN-запросов в момент запроса.
